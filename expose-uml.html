Introduction

En génie logiciel, la modélisation des besoins fonctionnels constitue une étape cruciale dans le cycle de développement d'un système. Le langage UML (Unified Modeling Language) propose pour cela le diagramme de cas d'utilisation, qui offre une vue synthétique des interactions entre les acteurs et le système.

Cependant, ce diagramme reste volontairement abstrait et visuel. Il montre quoi mais pas comment. Pour pallier cette limite, on recourt à la description textuelle des cas d'utilisation, une représentation complémentaire qui détaille de manière narrative et structurée le déroulement de chaque cas d'utilisation.

Cette description constitue un outil de communication essentiel entre les analystes, les développeurs et les parties prenantes du projet. Elle permet de valider la compréhension des besoins et sert de référence tout au long du développement.

Dans cet exposé, nous définirons d'abord ce qu'est une description textuelle et ses objectifs. Nous présenterons ensuite sa structure type, les différents scénarios qu'elle contient, puis nous illustrerons notre propos par plusieurs exemples concrets. Enfin, nous aborderons les bonnes pratiques à suivre pour rédiger des descriptions de qualité.

I. Définition et Objectifs

1.1 Définition

La description textuelle d'un cas d'utilisation est une représentation structurée (sous forme de fiche, tableau ou document) qui décrit de manière détaillée le déroulement des interactions entre un acteur et le système pour atteindre un objectif précis. Elle complète le diagramme de cas d'utilisation en apportant le niveau de détail nécessaire à la compréhension et à l'implémentation.

Contrairement au diagramme qui offre une vue d'ensemble graphique, la description textuelle adopte une approche narrative, étape par étape, qui explicite le dialogue entre l'utilisateur et le système.

1.2 Objectifs

Documenter les besoins fonctionnels : Elle formalise de manière précise ce que le système doit faire, sans ambiguïté. Chaque fonctionnalité est décrite avec ses conditions de déclenchement, son déroulement et ses résultats attendus.

Faciliter la communication : Elle sert de support d'échange entre les différentes parties prenantes du projet. Les analystes, développeurs, testeurs et clients peuvent s'y référer pour partager une compréhension commune.

Valider les exigences : En présentant la description au client ou aux utilisateurs finaux, on peut vérifier que l'équipe de développement a bien compris les besoins exprimés.

Guider le développement : Les développeurs s'appuient sur ces descriptions pour implémenter les fonctionnalités. Elles servent également de base pour la conception des tests.

Assurer la traçabilité : Chaque cas d'utilisation peut être relié aux exigences initiales du cahier des charges, permettant de vérifier que tous les besoins sont couverts.

II. Structure d'une Fiche de Description

Une fiche de description textuelle suit généralement un format standardisé comprenant plusieurs rubriques.

2.1 Identification du cas d'utilisation

Nom du cas d'utilisation : Un intitulé clair, généralement formulé avec un verbe à l'infinitif (ex : « Retirer de l'argent »).

Identifiant : Un code unique permettant de référencer le cas d'utilisation (ex : UC-001).

Version et date : Pour assurer le suivi des modifications.

Auteur : La personne responsable de la rédaction.

2.2 Description générale

Résumé : Une brève description en une ou deux phrases de l'objectif du cas d'utilisation.

Acteur principal : L'acteur qui initie le cas d'utilisation et en tire le bénéfice principal.

Acteurs secondaires : Les autres acteurs ou systèmes externes impliqués.

2.3 Conditions

Préconditions : Les conditions qui doivent être vraies avant le déclenchement du cas d'utilisation.

Postconditions : L'état du système après l'exécution réussie.

Déclencheur : L'événement qui provoque le lancement du cas d'utilisation.

2.4 Scénarios

Scénario nominal : La séquence d'étapes qui se déroule dans le cas normal.

Scénarios alternatifs : Les variantes possibles du déroulement normal.

Scénarios d'exception : Les cas d'erreur et la manière dont le système les gère.

2.5 Informations complémentaires

Règles de gestion : Les contraintes métier applicables.

Besoins spéciaux : Exigences non fonctionnelles (performance, sécurité).

Fréquence d'utilisation : Estimation de la fréquence d'exécution.

III. Les Scénarios

Les scénarios constituent le cœur de la description textuelle. Ils décrivent le dialogue entre l'acteur et le système sous forme d'une séquence d'étapes numérotées.

3.1 Le scénario nominal

Le scénario nominal représente le déroulement idéal du cas d'utilisation, celui où tout se passe comme prévu. Il décrit le chemin « ensoleillé » (happy path) menant à la satisfaction de l'objectif de l'acteur.

Caractéristiques : Il commence par une action de l'acteur principal, alterne entre actions de l'acteur et réponses du système, et se termine par l'atteinte de l'objectif sans erreur.

Conventions : Chaque étape est numérotée, rédigée au présent, avec une seule action par étape en précisant qui agit.

[Image 4] — Représentation du scénario nominal comme un chemin droit (flèches vertes).

3.2 Les scénarios alternatifs

Les scénarios alternatifs décrivent des variantes valides du scénario nominal. Ce sont des chemins différents mais tout aussi acceptables pour atteindre l'objectif.

Notation : On les numérote en référence à l'étape où ils divergent (ex : 3a, 3b pour des alternatives à l'étape 3).

[Image 5] — Représentation des scénarios alternatifs comme des embranchements (flèches bleues).

3.3 Les scénarios d'exception

Les scénarios d'exception gèrent les situations d'erreur ou les cas où l'objectif ne peut pas être atteint. Ils décrivent comment le système réagit face aux problèmes.

Notation : On utilise souvent un préfixe « E » (ex : E1, E2).

Exemples : Données invalides, ressource indisponible, droits insuffisants, timeout.

IV. Exemples Pratiques

Exemple 1 : Distributeur Automatique de Billets (DAB)
Cas d'utilisation : Retirer de l'argent
RubriqueDescription
IdentifiantUC-DAB-001
NomRetirer de l'argent
Acteur principalClient bancaire
Acteurs secondairesSystème d'information de la banque
RésuméPermet à un client de retirer des espèces depuis son compte via le DAB
PréconditionsLe DAB est opérationnel et approvisionné
DéclencheurLe client insère sa carte bancaire
Postconditions (succès)Le client a reçu les billets, son compte est débité

Scénario nominal :

ÉtapeActeurAction
1ClientInsère sa carte bancaire dans le DAB
2SystèmeLit la carte et demande le code PIN
3ClientSaisit son code PIN à 4 chiffres
4SystèmeVérifie le code PIN auprès du SI bancaire
5SystèmeAffiche le menu des opérations disponibles
6ClientSélectionne « Retrait d'espèces »
7SystèmePropose les montants prédéfinis et l'option montant libre
8ClientChoisit un montant (ex : 50 000 FCFA)
9SystèmeVérifie le solde et le plafond de retrait
10SystèmeDébite le compte et distribue les billets
11SystèmeÉjecte la carte et affiche un message de fin
12ClientRécupère sa carte et ses billets
Scénarios alternatifs :

8a. Le client choisit un montant libre : Le système affiche un écran de saisie → Le client saisit le montant → Retour à l'étape 9

Scénarios d'exception :

E1. Code PIN incorrect : Le système affiche une erreur et demande de ressaisir (3 tentatives max). Après 3 échecs, la carte est avalée.

E2. Solde insuffisant : Le système affiche « Solde insuffisant » et propose un autre montant.

Exemple 2 : Système de Réservation d'Hôtel
Cas d'utilisation : Réserver une chambre
RubriqueDescription
IdentifiantUC-HOT-001
NomRéserver une chambre
Acteur principalClient
Acteurs secondairesSystème de paiement en ligne
RésuméPermet à un client de rechercher et réserver une chambre d'hôtel en ligne
PréconditionsLe client est connecté à son compte
Postconditions (succès)La réservation est confirmée, la chambre bloquée

Scénario nominal :

ÉtapeActeurAction
1ClientAccède à la page de recherche
2ClientSaisit les critères (destination, dates, personnes)
3SystèmeRecherche les chambres disponibles
4SystèmeAffiche la liste des chambres avec tarifs
5ClientSélectionne une chambre et clique sur « Réserver »
6SystèmeAffiche le récapitulatif et demande confirmation
7ClientSaisit les informations de paiement
8SystèmeTraite le paiement et confirme la réservation
9SystèmeEnvoie un email de confirmation
Scénarios d'exception :

E1. Aucune chambre disponible : Le système affiche « Aucune disponibilité » et suggère des dates alternatives.

E2. Échec du paiement : Le système propose de réessayer ou changer de moyen de paiement.

Exemple 3 : Plateforme E-commerce
Cas d'utilisation : Passer une commande
RubriqueDescription
IdentifiantUC-ECO-001
NomPasser une commande
Acteur principalClient
Acteurs secondairesSystème de paiement, Service de livraison
RésuméPermet à un client de finaliser l'achat des articles de son panier
PréconditionsLe panier contient au moins un article
Postconditions (succès)Commande enregistrée, paiement effectué, livraison planifiée

Scénario nominal :

ÉtapeActeurAction
1ClientClique sur « Valider mon panier »
2SystèmeAffiche le récapitulatif et vérifie le stock
3ClientSélectionne l'adresse de livraison
4SystèmePropose les modes de livraison avec tarifs
5ClientChoisit un mode de livraison
6SystèmeCalcule le total et affiche les moyens de paiement
7ClientSélectionne un moyen de paiement et paie
8SystèmeEnregistre la commande et envoie la confirmation
Scénarios alternatifs :

7a. Paiement mobile (Orange Money) : Le système redirige vers le service → Le client confirme → Retour à l'étape 8

Scénarios d'exception :

E1. Article épuisé : Le système signale l'indisponibilité et propose des alternatives.

RubriqueDescription
IdentifiantUC-BIB-001
NomEmprunter un livre
Acteur principalAdhérent
Acteurs secondairesBibliothécaire (optionnel)
RésuméPermet à un adhérent d'emprunter un ou plusieurs livres
PréconditionsL'adhérent possède une carte valide et n'a pas atteint son quota

ÉtapeActeurAction
1AdhérentPrésente sa carte à la borne

Exemple 5 : Système de Gestion de Scolarité
Cas d'utilisation : S'inscrire à une année universitaire
RubriqueDescription
IdentifiantUC-SCO-001
NomS'inscrire à une année universitaire
Acteur principalÉtudiant
Acteurs secondairesService de scolarité, Service financier
RésuméPermet à un étudiant d'effectuer son inscription administrative
PréconditionsL'étudiant est admis ou autorisé à se réinscrire
Postconditions (succès)L'inscription est validée, la carte d'étudiant peut être éditée

Scénario nominal :

ÉtapeActeurAction
1ÉtudiantSe connecte au portail d'inscription
2SystèmeVérifie l'identité et les droits d'inscription
3SystèmeAffiche le formulaire pré-rempli
4ÉtudiantVérifie et met à jour ses informations
5ÉtudiantTéléverse les pièces justificatives
6SystèmeVérifie la complétude et calcule les frais
7ÉtudiantChoisit le mode de paiement et paie
8SystèmeValide l'inscription et génère le certificat
9ÉtudiantTélécharge son certificat de scolarité
Scénarios alternatifs :

7a. Étudiant boursier : Le système vérifie le statut et applique l'exonération partielle ou totale.

7b. Paiement échelonné : Le système génère un échéancier → L'étudiant paie la première tranche.

Scénarios d'exception :

E1. Dossier incomplet : Le système liste les documents manquants, l'inscription reste en attente.

E2. Étudiant non autorisé : Le système affiche le motif du blocage et invite à contacter la scolarité.

V. Bonnes Pratiques et Conseils

5.1 Règles de rédaction

Utiliser un vocabulaire cohérent et précis. Chaque terme métier doit être défini dans un glossaire et utilisé de manière uniforme. Par exemple, si l'on parle de « client » dans un cas d'utilisation, on ne doit pas utiliser « utilisateur » pour désigner la même entité dans un autre.

Numéroter systématiquement les étapes. La numérotation facilite les références, notamment pour les scénarios alternatifs et d'exception qui renvoient à des étapes précises.

Rédiger au présent de l'indicatif et à la voix active. On écrit « Le système affiche le message » plutôt que « Le message sera affiché par le système ».

Une seule action par étape. Chaque étape doit décrire une action atomique. « Le client saisit son identifiant et son mot de passe puis clique sur connexion » devrait être divisé en trois étapes.

Identifier clairement l'auteur de chaque action. Chaque étape doit préciser qui agit : l'acteur ou le système.

5.2 Niveau de détail

Adapter la granularité au contexte. Le niveau de détail dépend de la complexité du cas d'utilisation et des besoins des lecteurs. Une description destinée aux développeurs sera plus précise qu'une description pour validation client.

Rester au niveau métier. La description ne doit pas anticiper les choix techniques. On écrit « Le système enregistre les informations » et non « Le système exécute une requête INSERT dans la table Clients ».

Se concentrer sur le "quoi" et non le "comment". La description exprime ce que le système doit faire, pas comment il le fait techniquement.

5.3 Gestion des relations entre cas d'utilisation

Décrire les inclusions (≪include≫). Indiquer clairement à quelle étape l'inclusion se produit : « Le système exécute le cas d'utilisation "S'authentifier" (UC-AUT-001) ».

Décrire les extensions (≪extend≫). Préciser la condition de déclenchement et le point d'extension.

Maintenir des références croisées. Chaque fiche doit mentionner les cas d'utilisation liés pour faciliter la navigation.

Éviter la redondance. Factoriser les comportements communs dans des cas d'utilisation inclus.

5.4 Validation et qualité

Impliquer les parties prenantes. La description doit être relue et validée par les utilisateurs métier et le client.

Vérifier la cohérence avec le diagramme. Tout cas d'utilisation présent dans le diagramme doit avoir sa description, et inversement.

Assurer la traçabilité. Chaque cas d'utilisation doit pouvoir être relié aux exigences fonctionnelles du cahier des charges.

Tester les scénarios. Dérouler mentalement chaque scénario pour vérifier qu'il est complet et cohérent.

5.5 Erreurs fréquentes à éviter

Décrire l'interface utilisateur plutôt que l'interaction. Ne pas spécifier les éléments d'interface (« Le client clique sur le bouton vert en haut à droite »).

Mélanger plusieurs cas d'utilisation. Chaque fiche doit correspondre à un seul objectif utilisateur.

Oublier les cas d'erreur. Les scénarios d'exception sont souvent négligés alors qu'ils représentent une part importante du comportement réel.

Négliger les préconditions et postconditions. Ces éléments sont essentiels pour délimiter le périmètre du cas d'utilisation.

Rédiger des scénarios trop longs. Un scénario de plus de 15-20 étapes devient difficile à suivre. Il faut envisager de décomposer.

Conclusion

La description textuelle des cas d'utilisation constitue un outil indispensable dans la boîte à outils de l'analyste et du concepteur de systèmes d'information. En complétant le diagramme de cas d'utilisation, elle apporte le niveau de détail nécessaire pour passer de la vision globale à la compréhension fine des interactions entre les acteurs et le système.

À travers cet exposé, nous avons vu que la description textuelle suit une structure rigoureuse comprenant l'identification du cas, les acteurs impliqués, les conditions d'exécution et surtout les scénarios qui détaillent le déroulement pas à pas. La distinction entre scénario nominal, scénarios alternatifs et scénarios d'exception permet de couvrir l'ensemble des situations possibles.

Les cinq exemples présentés, issus de domaines variés (bancaire, hôtelier, e-commerce, bibliothèque et universitaire), illustrent la versatilité de cet outil qui s'adapte à tout type de système. Quelle que soit la complexité du projet, la méthode reste la même.

Enfin, le respect des bonnes pratiques de rédaction garantit la qualité et l'utilité des descriptions produites. Un vocabulaire précis, un niveau de détail approprié, une validation par les parties prenantes et une maintenance régulière sont les clés d'une documentation efficace.

La description textuelle s'articule avec d'autres diagrammes UML pour offrir une vue complète du système. Le diagramme de séquence peut être dérivé directement du scénario nominal, tandis que le diagramme d'activité peut illustrer les flux avec leurs branchements. Cette complémentarité fait la force de la modélisation UML.

En maîtrisant la description textuelle des cas d'utilisation, le futur ingénieur ou chef de projet dispose d'un moyen efficace pour capturer, communiquer et valider les besoins fonctionnels, première étape vers la réussite de tout projet de développement logiciel.
